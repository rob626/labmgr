What is the process flow for submitting an Open Lab?

1) Create your lab virtual machine within your development environment. It would be best to use VMware Workstation 10 with a Hardware Compatability mode of Workstation 10 or Workstation 9.

2) Create your lab PDF instructions. Make sure to use the Lab template (see link on left) and ensure quality as these are customer facing.

3) Complete your local testing. Before uploading your lab to the reference machine, your lab should be completely tested and validated within your test environment.

4) Upload your VM(s) and PDF to reference machines. Work with your Track Lead to schedule time on your track reference machine. Your track Lead will provide FTP access information (for the uploads) and RDP access information for validation.

    • 4a) Create a directory C:\Labs\<session-id> (For complex cases, see notes below)
    • 4b) Add your lab instructions C:\Labs\<session-id>_IC16.PDF For example, C:\Labs\abc1234_IC16.PDF
    • 4c) Create a directory for your VM C:\Labs\<session-id>/<system-label>/. For example, C:\Labs\abc1234\server\. No spaces in the names or paths please.
    • 4d) Upload your expanded VM into C:\Labs\<session-id>/<system-label>/.
    • 4e) If you have more than 1 VM for your session, create addition <system-label> folders and upload the VMs. For example, C:\Labs\abc1234\server\ and C:\Labs\abc1234\workstation\.

5) Validate your VM. Connect to the reference machine using Remote Desktop (RDP) and ensure your VM works correctly on the reference machine. The reference machine uses the same hardware and setup used at the conference. This process should be simple as the initial testing and validation should occur in your local development environment.

    • RDP Host - this is the host name for the reference machine
    • RDP Username - IBM_USER
    • RDP Password - web1sphere

6) Validation complete. Once the validation process is complete and your are ready to submit:

    • 6a) Shutdown the VM
    • 6b) Create a snapshot (named startlab) for the state the VM needs to be started to being the lab
    • 6c) Change the VM name (in Workstation, VM > Settings > Options > Virtual machine name) to be the system label
    • 6d) Exit VMware Workstation
    • 6e) Rename your session directory to include a _DONE at the end. For example, C:\Labs\<session-id>_DONE

7) Fill out a lab submission form. Open the SUBMIT.PDF file in C:\Labs and fill it out. Save the completed form into your session directory with the name of C:\Labs\<session-id>\<session-id>_SUBMIT.PDF. There will be a separate video on the form filling process available on the left.

8) Email the lab team and Track Lead. Send an email to the Lab Team (email alias Edu_SPLL_Submission-Tiv-WW) and your Track Lead letting them know that you have completed the submission process.


Complex Cases:. In the simple case, there is only VM or set of VMs for a single lab session. The complex cases are for when there are multiple labs share a VM or set of VMs. In such cases, the lab directory should include multiple session ids. For example:

    • C:\Labs\abc1234_abc2345\ - indicates that this set of VMs will be used for both lab abc1234 and lab abc2345. So, to create a list, separate the session ids with an underscore ( _ ).
    • C:\Labs\abc1234-abc1236\ - indicates that this set of VMs will be used for labs abc1234, abc1235, and abc1236. So, to create a range, separate the session ids with a dash ( - ).
    • C:\Labs\abc1234-abc1236_abc1239\ - indicates that this set of VMs will be used for labs abc1234, abc1235, abc1236, and abc1239. So, you can combine the underscore and dash.

Complex Cases snapshots. If you are using a VM or set of VMs for multiple labs, you may need multiple starting points. If not, simply use 'startlab' as the snapshot name. If multiple snapshots are needed, please use 'startlab_<session-id>. For example:

  • startlab_abc1234 and startlab_abc1235 - 2 snapshots which offer unique starting points.
  • startlab - A single snapshot could be shared by multiple lab sessions.

    • Lab image(s)

You need to upload both your lab image and the lab instructions by Friday, January 8th.  Both need to be uploaded to your reference machine.


My lab only needs a browser, do I still need a VM?

Yes.

For various reasons, we need to have every lab session work through a VM and not use the host browser. If your lab only needs a browser, there will be a base Windows 7 VM available on every lab machine. Feel free to let the lab team know that you prefer to just use the shared base VM instead of uploading your own.

If you choose to use the common base, copy the VM from the C:\Labs\shared\ folder and verify that the VM works for your lab.


What is the testing process?

The lab team will only verify that the VM(s) submitted start correctly from the snapshot.

We strongly encourage you to find a team mate or coworker to work through the lab independently of the author. This person should focus not just on functional testing (to make sure the lab works exactly as written), but also for language / grammar issues.

What are the expectations for the lab write-ups?

The expectations are that the labs instructions are complete and professional. Remember, these are being presented to IBM customers. Please make sure they are:

    • Polished / Proofed
    • Spell checked
    • Gramatically correct
    • Technically correct
    • Verified - please make sure someone goes through the final instructions before submitting

What are some best practices for the lab write-ups?

Here are some best practices to consider when creating your lab instructions:

    • Start with lab template - or at least copy the cover page into your instructions
    • Include start up instructions - what does the student need to do to get started, including logging in and starting servers
    • List all necessary userids / passwords at the start of the lab
    • Consider using a single password to make it easier for the student
    • Include an Overview which describes the lab from a high level
    • Describe the lab objectives
    • End the exercise with a summary where you review what you just did
    • For labs that will be published to PartnerWorld for business partners and the Technical Enablement Portal for IBMers, make sure the lab is written so it functions well in a self-paced, stand-alone environment. There should be no reference to proctors or the InterConnect environment. Avoid specific machine logins and configurations that are specific to the conference.

What are the details for lab prints?

Please limit the lab instructions to no more than 100 pages. The page limit is due to printer limitations associated with print jobs.

A limited number of instructions will be printed and then shared by the attendees.

Licensing issues... what do I need to worry about?

    • Non-IBM software: Any software on the virtual machine that is not directly provided by the operating system or by IBM (such as third party software, freeware, shareware, and even open source) will need to be appropriately licensed.

    • Operating systems for the guest: We need to be very careful that we are covered for any operating systems that are used by our guest VMs.

IMPORTANT: if there is ANY question about licensing issues, please work with your legal team or the Software Licensing Group to ensure that you are compliant.

For additional information or questions, see this Software Group Software License Management Wiki or contact Tim Robinson (Timothy M Robinson/Fairfax/IBM).


Where do I get base images?

Several base virtual machines are available in the 'shared' directory in C:\Labs\shared\. These include a Windows 7 VM as well as several versions of Redhat EL 6.x VMs.


What version of VMware will be used?

VMware Workstation v10 will be used on the conference machines. This also means that if the image is exported from Fusion or ESX, you need to ensure that the Hardware compatability is set to no higher than Workstation 10. We find that version 9 or version 10 work best.

You should also install an up to date version of VMware Workstation v10 Tools on your VMs. If you are moving from another hypervisor (such as ESX or Fusion), the version of Tools is different.

NAT or Bridged?

The preferred and safest approach is to configure your VM to use a NAT'ed network. If you need to use Bridged, that is fine as well. In some environments, there may be a limited number of IP addresses, so Bridged VMs can experience issues. Therefore, if NAT works for your needs, we recommend using NAT. Of course, Host-Only is best if no network access is needed.


VMware: Copy vs Move?

The lab team will assume COPY unless you let us know otherwise.

When virtual machines are moved to a new environment, Workstation typically asks if the VM is Copied or Moved. By default, Copy is used when deploying virtual machines to the lab systems.

The main difference between the way Workstation handles COPY versus MOVE is that a COPY will create a new MAC address for the virtual network card. The MOVE keeps the existing MAC address. In some rare cases a MOVE is necessary in order to keep the original MAC address.

To zip or not to zip?

The final deliverable on the reference machine needs to be unzipped. But, to upload your VM(s), it is usually more reliable and faster to zip before doing your network transfer.
